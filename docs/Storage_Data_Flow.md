## Storage Comparison Table

| Storage Layer | Write Speed | Read Speed | Persistence | Use Case |
|--------------|-------------|------------|-------------|----------|
| **Component State** | Instant | Instant | Session only | UI updates, temporary state |
| **Context State** | Instant | Instant | App lifetime | Shared state across components |
| **Service Layer** | Instant | Instant | Page lifetime | Audio control, in-memory cache |
| **Socket.IO** | 20-100ms | 20-100ms | Connection | Real-time sync between users |
| **MongoDB** | 50-200ms | 50-200ms | Permanent | Playlist, participants history |
| **Supabase S3** | 1s per MB | 100-500ms | Permanent | Music file storage |

## Key Architectural Decisions

### âœ… What Gets Persisted to MongoDB
- Music playlist (track metadata + file URLs)
- Participant join/leave history with timestamps
- Session metadata (group_id, session_type)
- Music controller assignment (who is DJ)

### âŒ What Does NOT Get Persisted
- Real-time playback state (playing/paused/position)
- Microphone mute states
- WebRTC connection states
- Audio level meters

### ğŸ”„ Deduplication Strategy
- **Frontend checks** for duplicates BEFORE adding to context
- **On session load** from MongoDB, deduplicates playlist
- **On playlist updates** from other users, deduplicates
- Comparison: File URL first, then title+artist combo

### âš¡ Performance Optimizations
1. **Socket.IO autoConnect: false** - Register handlers before connecting to avoid race conditions
2. **Duplicate checks** - Prevent adding same track multiple times
3. **Field mapping** - Handle both snake_case (MongoDB) and camelCase (Frontend)
4. **Fallback session data** - Allow basic functionality even if backend is unavailable
